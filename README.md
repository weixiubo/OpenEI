# OpenEI

OpenEI 是面向具身机器人的通用多模态智能体运行时与任务执行框架。项目目标是把多源感知输入、模型推理、任务语义、技能调度和硬件控制统一到一条可调度、可执行、可观测的机器人任务闭环中。

当前仓库保留了一套可在真实机器人和模拟环境中运行的端侧闭环实现，覆盖语音交互、音频感知、任务解析、技能序列生成、硬件通信、异常降级和启动自检等关键工程环节。

## 核心能力

- 多模态感知接入：支持语音输入和音频时序特征接入，框架层面通过统一任务表示承接不同感知源。
- 模型推理与意图解析：将开放式用户输入转化为包含目标、参数、执行边界和安全确认的结构化任务意图。
- 任务中间表示：把外部输入、上下文、执行约束和技能调用条件统一为可传递的任务状态。
- 技能库与序列调度：将机器人原子能力抽象为可注册、可查询、可组合的技能单元，并根据任务约束生成连续执行序列。
- 端侧执行运行时：提供运行档位、启动自检、真实/模拟传输切换、设备自动检测和执行反馈。
- 硬件适配与异常恢复：通过硬件适配层屏蔽底层通信细节，在硬件不可用、网络异常或感知失败时进入降级路径，保证任务链路不中断。

## 快速开始

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 配置环境变量

复制 `.env.example` 为 `.env`，填入模型服务、语音服务和硬件通信配置。

### 3. 启动

Linux / Orange Pi 推荐：

```bash
bash scripts/run_demo.sh
```

Windows 本地验证：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run_demo.ps1
```

手动启动：

```bash
python main.py --profile demo --transport auto --recording-mode smart_vad
```

## 运行模式

- `--profile demo`：演示档位，启用启动自检、状态面板和高风险任务确认。
- `--profile dev`：开发档位，保留更直接的本地调试体验。
- `--transport auto`：优先连接真实硬件，失败后自动切换模拟执行。
- `--transport real`：强制真实硬件模式。
- `--transport sim`：强制模拟执行模式。
- `--recording-mode smart_vad`：默认智能录音模式。
- `--no-tts`：禁用语音播报，仅输出文本反馈。

## 目录说明

- `config/`：运行配置、音频参数、硬件通信参数和运行档位。
- `core/`：感知特征处理、节拍跟踪和技能序列调度逻辑。
- `dance/`：历史命名保留的机器人技能库、控制器和串口驱动模块。
- `voice/`：录音、语音识别、语音合成、意图解析和交互运行时。
- `data/`：机器人技能元数据。
- `tests/`：单元测试和回归测试。
- `docs/`：当前说明、演示排练和历史实验材料。
- `archive/`：历史代码快照和实验分支。

## 工程建议

- 先用 `--transport sim` 验证完整任务链路。
- 再切换 `--transport auto` 检查硬件适配和串口自动检测。
- 现场网络不稳定时，对话能力会降级为固定反馈，不影响任务执行主链路。
- 感知模块不可用时，系统会切换到降级任务源，避免端到端流程中断。
